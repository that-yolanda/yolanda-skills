# README SKILL

为项目根目录创建或更新 `README.md`。

README 代表“当前真相 + 已确认规划”。

## 输出结构

严格使用以下结构：

```md
# 项目名称
一句话介绍该项目

## 项目脚本
%%例如初始化，调试，运行，测试...%%
### 项目初始化

### 项目本地运行

### 项目启动

### 项目调试


## 项目功能（含规划）
%% 包含规划/标注是否完成%%
### 模块 xxx
- [ ] **功能1名称xxxx**: 功能概述
- [ ] **功能2名称xxxx**: 功能概述

## 项目结构
%% 文件树形式，仅存放文件夹和核心文件，并在每一项后追加用途说明 %%

---
更新日期:xxxx-xx-xx
```

## 执行流程

1. 快速扫描项目结构、配置、脚本、源码、已有 README、文档和路线图。
2. 判断自己是否已理解项目定位、模块规划和核心功能。
3. 若只能看出项目框架，无法确认目标或规划，先向用户提问。
4. 整理项目名称、简介、脚本、当前功能、已确认规划和目录结构。
5. 创建或更新根目录 `README.md`。

## 内容规则

- 基于仓库事实和已确认信息写内容，不要臆造。
- README 必须同时反映：
  - 当前真相：代码、配置、文档中已能确认的内容
  - 已确认规划：用户明确说明或文档明确记录的后续方向
- 信息不足时先提问；仍无法确认时写“待补充”或“暂无”。
- “项目脚本”只写仓库中真实存在的命令。
- “项目功能（含规划）”必须标注状态：
  - 已完成使用 `- [x]`
  - 规划中使用 `- [ ]`
- 功能分组按业务模块、产品模块、服务模块、页面模块等组织，不按技术栈堆砌。
- “项目结构”使用文件树，只保留核心目录和关键文件，忽略构建产物与噪声文件。
- “项目结构”中的每个目录或文件都要在同一行追加简短用途说明，例如 `# 主进程入口`。
- “更新日期”使用当前本地日期，格式必须是 `YYYY-MM-DD`。
- 保持通用，不绑定特定框架或目录约定。

## 项目脚本写法

- `项目初始化`：安装依赖、环境准备、初始化命令。
- `项目本地运行`：开发态运行、本地预览。
- `项目启动`：正式启动、生产启动、打包后启动。
- `项目调试`：lint、format、build、check、test、typecheck 等命令。

没有对应命令时，保留标题并写“暂无”。

## 项目功能写法

- 先看现有模块、入口、接口、脚本或页面，再总结功能。
- 若存在 TODO、规划文档、占位实现、接口草稿或模块空壳，可写为规划项，但前提是这些规划可确认。
- 每个模块只保留高价值功能点。
- 功能描述控制在一句话内。
- 当代码仍是框架阶段，不要把空目录写成已完成能力，应优先记录总体规划和核心功能规划。

## 项目结构写法

使用文件树形式，只保留：

- 顶层核心目录
- 关键入口文件
- 代表性配置文件
- 每一项都追加 `# 用途说明`

## 更新策略

- 根目录不存在 `README.md` 时直接创建。
- 根目录已存在 `README.md` 时，保留有价值信息，但整体改写为本 skill 规定的结构。
- 初始化阶段先快速浏览项目；若无法确认总体规划或核心功能，先向用户提问。
- 更新阶段在模块或功能新增、修改、删除后同步更新 README。
- 确保 README 与项目实际及已确认规划保持一致。
- 对已取消、已替换、已落地的规划及时调整描述和状态。
- 除非用户明确要求，否则不要写成营销文案、英文模板或超长说明。
- 除非用户明确要求，否则不要新增额外说明文件。
