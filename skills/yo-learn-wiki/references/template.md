# 内容规范

准入标准、写作质量和页面结构的唯一定义处。

## 封面选图

根据主题领域自动选择封面图，无需询问用户。

```bash
ls "{cover_dir}"/*.jpg 2>/dev/null | xargs -n1 basename
```

`{cover_dir}` 从 EXTEND.md 读取。根据文件名语义选择与页面主题最匹配的图片（如 `bot.jpg` → 人工智能，`terminal.jpg` → 代码，`brain.jpg` → 方法论/思维）。若无法匹配，选最通用的图片。

## 准入标准

专题创建和内容录入共用同一套核心标准，评估粒度不同：

| 标准 | 专题级（创建专题时） | 内容级（ingest 时） |
|------|---|---|
| **任务可服务** | 这个专题服务于什么 AI 任务场景？让 agent 执行某类任务时产出更好的结果 | 这条内容包含可操作的方法、判断标准或原则？能让 agent 在具体场景下做出更好的决策 |
| **可复用可积累** | 这个领域能持续积累，形成越来越完整的体系（不是一次性的知识块） | 不是临时信息（调试日志、新闻、个人备忘），能在未来反复使用 |
| **有增量** | 知识库中尚无此领域的体系化沉淀 | 比已有页面提供了新认知（新观点、新案例、新方法论） |

三项必须同时满足。不满足时说明原因，拒绝入库。

### 准入示例

**应该入库**：思维框架（AI 辅助创作方法论）、可复用流程（SEO 优化体系）、设计原则（Skill 设计分类学）、认知模型（行为学洞察）
**不应入库**：具体工具操作教程、新闻公告、调试日志、某个 API 的使用笔记、零散技术备忘

### 判断结果

| 结果 | 处理 |
|------|------|
| 三项都满足 | 继续 |
| 碎片太小但值得保留 | 归入已有页面补充 |
| 不满足 | 拒绝，说明原因 |
| 已有内容覆盖 | 拒绝，指出现有页面 |

## 写作质量指南

核心目标：**人和 AI 读完都能直接用。**

### 结构与连贯

- **用 heading 层级组织内容**：按步骤（## 步骤1 → ## 步骤2）、按模块（## 核心概念 → ## 关键机制 → ## 实践要点）、按场景（## 场景A → ## 场景B）等。具体用哪种由内容和专题约定决定
- **追加内容时保持连贯**：新增内容不是独立段落，而是融入现有逻辑流。更新后整篇内容应是递进/并列/总分结构，而非拼凑
- **整体结构优先于局部**：更新时先评估新内容对整体结构的影响，再决定插入位置和方式

### 内容质量

- **用自己的语言**，不搬运原文，不嵌入大段引用或代码块。确保不看源材料也能理解和应用
- **优先写可操作的判断**（"当 X 时选方案 A，当 Y 时选方案 B"），而非笼统描述（"要根据实际情况选择"）
- **包含具体标准**：有阈值给阈值，有步骤给步骤，有清单给清单
- **说清楚边界**：适用什么场景、不适用什么场景
- **AI 可执行**：agent 读完知道「什么时候用」「怎么用」「用什么步骤」

### 更新时的整体评估

更新已有页面时，不只是追加内容，需要从整体结构上考虑：

1. **主题范围**：新内容是否改变了页面的主题范围 → 更新 `description`
2. **话题维度**：新内容是否引入了新的话题维度 → 更新 `tags`
3. **逻辑顺序**：新内容是否需要调整页面的逻辑顺序 → 重排 heading
4. **结构适用性**：新内容是否让现有 heading 结构不够用 → 重构内容层级
5. **文件命名**：新内容是否改变了页面的核心定位 → 考虑重命名文件

## 页面结构

### 专题索引页

文件名格式 `00-{专题名}.md`（如 `SEO优化/00-SEO优化.md`），确保在 base 视图中始终排在专题内第一个。

Frontmatter：

```yaml
---
tags: [{主题}, {主题}, ...]
description: "{50字内：这个专题解决什么问题}"
cover: "[[{image_name}.jpg]]"
---
```

正文必须包含 `知识路径`（有序的知识页列表），其余 section 由内容自然决定。常见 section：概览、核心原则、页面约定等。

当索引页包含「页面约定」section 时，该专题下的知识页应遵循该约定。

### 知识页

文件名格式：`{序号}-{主题名}.md`。

Frontmatter：

```yaml
---
tags: [{主题}, {类型}]
description: "{一句话：这个知识点是什么，解决什么问题}"
cover: "[[{image_name}.jpg]]"
---
```

正文结构由内容和专题约定决定，用 heading 层级按逻辑组织（步骤/模块/场景等）。末尾统一包含：

```markdown
## 更新日志

- [t] YYYY-MM-DD：{增量描述} — [[源文件名]]

## 关联

- [>] [[页面名]] — 关系描述
```

## Tags 约定

- **主题标签**：精简、不重复。用最核心的词概括页面所属领域（如 `SEO`，不需要再加 `搜索优化`）
- **类型标签**：补充一个类型说明，帮助快速识别知识性质
- **专题索引页**：只打主题标签（如 `tags: [SEO]`）
- **知识页**：主题标签 + 一个类型标签（如 `tags: [SEO, 方法论]`）

常用类型标签：`方法论`、`技术方案`、`工具`、`操作手册`、`思维模型`、`商业模型`

## 全局规则

- `description`：专题索引 ≤50 字，知识页一句话概括主题和用途
- `## 更新日志`：每次更新追加一条，必须说明增量或矛盾，不改写历史条目
- `## 关联`：优先链接同专题内的前后页面，再链接跨专题页面
- 所有链接使用 Obsidian wikilink 语法 `[[文件名]]`
- 页面文件名和专题目录名使用中文
