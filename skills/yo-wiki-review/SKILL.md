# yo-wiki-review

定期扫描原子库，将已验证的原子提炼到知识库和用户画像。

`$WIKI_DIR` 是系统环境变量，指向知识库根目录。下文所有路径通过 `$WIKI_DIR` 引用。首次使用若未配置，按 [references/first-time-setup.md](references/first-time-setup.md) 引导用户设置。

## User Input Tools

当本 skill 需要提示用户时，按此工具选择规则（优先级顺序）：

1. **优先使用当前 agent runtime 内置的用户输入工具** — 如 `AskUserQuestion`、`request_user_input`、`clarify`、`ask_user`、`questionnaire` 或等效工具。
2. **回退到纯文本**：若无此类工具，输出编号的纯文本消息，让用户回复选择的编号/答案。
3. **批量规则**：若工具支持单次调用多个问题，将所有适用问题合并为一次调用；若仅支持单问题，按优先级逐个提问。

下文对 `AskUserQuestion` 的具体提及均为示例 — 其他 runtime 按规则替换为本地等效工具。

## 必须阅读

- `$WIKI_DIR/使用说明.md` — 跨层权限和 Skill 分工边界
- `$WIKI_DIR/知识库/使用说明.md` — 专题定义、文档格式、专题生命周期
- `$WIKI_DIR/原子库/使用说明.md` — 原子字段和 status 定义
- `$WIKI_DIR/我的画像/使用说明.md` — 画像更新规则
- 所有已有专题文档的 frontmatter `description` — 辅助判断原子归属

## 禁止行为

- 新增原子、修改原子原始内容、删除原子
- 改判原子录入时的 `type` / `confidence`
- 直接编辑 `atoms.jsonl`
- 覆盖知识库文档中已有的条目

## 执行流程

1. 确认评估范围：评估 status 为`null`(新入库，首次评估）与 `pending` 的原子信息（条件不足，无法判断是否值得录入）
2. 逐条判定：对范围内的每条原子，已有足够同类事实支撑 → 进入第 3 步，已被后续事实否定 → 标记待废弃不提炼，仍然信息不足 → 保持 `pending` 跳过
3. 判定归属：读每个专题的 `判断标准` 和对应文档的 `description`，按语义判定。有归属 → 进入第 5 步，无匹配但有价值 → 进入第 4 步
4. 专题新增：按 `$WIKI_DIR/知识库/使用说明.md` 专题生命周期的流程提议新增。用户确认前不强行塞入已有专题
5. 写入知识库：按 `$WIKI_DIR/知识库/使用说明.md` 文档格式的规范，将原子追加到对应专题文档的对应小节，更新 frontmatter 的 `updated` 和 `atom_count`
6. 更新画像：按 `$WIKI_DIR/我的画像/使用说明.md` 的规则，将提炼过程中对用户的新认识更新到对应文件
7. 回写原子状态：按 `$WIKI_DIR/原子库/使用说明.md` 的 status 定义，通过 `yo wiki update` 命令更新原子状态。提炼时发现原子 `type` / `tags` 分类错误，先修正再提炼，不在知识库文档中手工修改条目

## 收尾检查

- [ ] 确认所有待提炼原子已写入知识库
- [ ] 确认专题文档 frontmatter 已更新
- [ ] 确认原子 `status` 已回写
- [ ] 确认画像文件已更新
- [ ] 如有新增专题，已按专题生命周期完成创建
- [ ] 如有无匹配专题的原子且用户未确认，已标记待处理
