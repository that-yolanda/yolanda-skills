# 原子库使用说明

本文指导将用户输入的任意信息提取整理并记录到知识库原子库中。

## 核心原则
- 只记录可沉淀/有价值的资产，不记录无价值的咨询
  - 有价值：可以日后被使用的资产，例如：用户的偏好，方法论，某个技巧，案例，有价值的观点等
  - 无价值：例如：新闻，心情，动态等
  - 判断标准：这个内容下个月还能不能用的上
- 只提取原子信息，要点，不记录大量的原文，原文通过链接
- 永不删除：原子通过 `status` 字段管理生命周期，不物理删除

## 可用命令

命令格式：yo wiki [command] [options]
帮助说明：所有命令支持 -h 查看说明
查询记录：yo wiki search [keyword] [options]
新增记录：yo wiki add [record]

## 字段格式

| 字段         | 类型       | 说明                                                        |
| ---------- | -------- | --------------------------------------------------------- |
| id         | string   | 格式：{年}-{月}-{序号}，如 2024-04-001                             |
| knowledge  | string   | 提炼后的知识点陈述句                                                |
| original   | string   | 截取原文片段（最多200字）                                            |
| url        | string   | 外部链接                                                      |
| author     | string   | 知识点的原始提出者（见下文说明），如 `Yolanda`、`@dontbesilent`、`OpenAI`     |
| date       | string   | 发布日期，YYYY-MM-DD                                           |
| tags       | string[] | 话题标签（见下文说明）                                               |
| type       | string   | 类型（见下文说明）：principle，method，case，anti-pattern，tool，insight |
| confidence | string   | 价值初步判定：high / medium / low                                |
| status     | string   | 记录状态（见下文说明）： null / pending / achieved / discorded        |

## 字段说明
### 话题标签：tags
从以下中选择（可按自身领域调整）

| 分类   | tags                                        |
| ---- | ------------------------------------------- |
| 认知层面 | 商业模式，思维模式，案例与复盘                             |
| 实战能力 | AI与工具，内容创作，自媒体运营，产品运营，用户画像，需求挖掘，对标与模仿，提示词技巧 |

### 日期：date
- 知识点原文的发布日期
- 用户自己的经验/实验以实际发生日期为准
- 不可考证的按实际插入时间

### 作者：author
- 定义：`knowledge` 字段陈述的知识点的原始提出者，回答"这个观点/方法/案例是谁的"
- 用户自己的经验、实验、观点 → 你的名字
- 个人博主 → 可辨识的人名或常用 ID，如 `dontbesilent`、`Paul Graham`。不加平台前缀（如 `@`）
- 机构/公司官方内容 → 机构名，如 `OpenAI`、`Anthropic`
- 多人讨论中某人的观点 → 具体人名
- 用户综合多个来源后自己得出的结论 → 你的名字，原始素材各自一条原子标原作者
- 不可考证 → 留空
- 不记录：灵感来源（某个人的话启发你想到的但不是他直接说的），转述来源（你从别人那里听来的二手信息）
- 判断标准：是否是该知识点的直接、原始表述者

### 类型：type 
- principle（原则）- 可复用的原则、规律（解释 WHY）
  - 定义：长期成立的规律、判断标准、底层认知。
  - 记录：例如：商业模式决定内容策略；价格决定客户质量；先有需求，后有产品
  - 不记录：今天有效的运营技巧，平台规则变化，未验证的观点，某个人的主观看法
  - 判断标准：三年后大概率仍然成立。

- method（方法）- 可执行的方法（解释 HOW）
  - 定义：可执行的步骤、流程、工作流
  - 记录：例如：做内容前先拆用户问题；先验证需求再开发；先做 Workflow 再做 Agent。
  - 不记录：为什么这样做（这是 principle），单纯的经验总结，没有执行步骤的建议
  - 判断标准：能否写成第一步、第二步、第三步。

- case（案例）- 真实案例（解释 WHAT HAPPENED）
  - 定义：真实发生过的案例、实验、结果。
  - 记录：客观发生过的事实，例如：我用 8 美金 token 做出了选题 Skill；我用这个方法涨粉 1000；某 SaaS 月收入达到 2 万。
  - 不记录：猜测，预测，分析结果，主观感受，没有结果的尝试
  - 判断标准：是否真实发生过？是否有明确结果？

- anti-pattern（反模式）- 错误模式（论证 WHAT FAILS）
  - 定义：经常出现、但会导致坏结果的错误模式。
  - 记录：什么事做了就容易失败、踩坑。例如：产品未验证就开发；一开始设计复杂架构；过早优化。
  - 不记录：单次失败，偶然失误，运气不好导致的问题
  - 判断标准：是否属于重复出现的错误模式。是否具有借鉴意义。

- tool（工具）- 可复用资产（解释 WHAT TO USE）
  - 定义：利用工具、框架、系统或架构解决问题的可复用经验。
  - 记录：工具的关键设计思想、架构方案、实现方案、配置方案、最佳实践；从 Prompt、Skill、Agent、框架、技术文章中提炼出的可复用经验。例如：中文全文检索需要额外分词器（例如jieba）支持，否则检索效果较差；复杂 Agent 采用 Plan → Execute → Verify 结构。
  - 不记录：Cursor 很强；Claude 很好用；某工具最近很火；工具推荐；工具新闻；工具评价；功能介绍；完整 Prompt；完整 Skill；完整 Agent；完整 Workflow。
  - 判断标准：是否总结出了可复用经验；是否包含具体做法而非评价；脱离具体工具名称后是否仍然有价值。

- insight（洞察）- 洞察、观点（解释 WHAT MIGHT BE TRUE）
  - 定义：未经证实的事，不一定正确，只代表某个人或某些人的看法，但具有一定价值。
  - 记录：有价值但尚未形成规律的观察、判断、猜想。例如：AI 搜索会逐步蚕食传统搜索。播客在小红书的商业化效率可能偏低。Agent 最终竞争的是上下文资产。
  - 不记录：情绪表达，吐槽，阴谋论，无依据判断，已经验证的规律（应该归类为 principle）
  - 判断标准：有一定证据支撑。具备启发价值。尚未被充分验证。

### 状态：Status
- 状态流向：null → pending → achieved / discorded
- 刚入库字段，该字段为空
- 系统周期性会进行审核，结合现有知识库/我的画像中的内容，判断原子记录的价值：
  - 条件不足，则标注为 pending，等待后续积累更多原子信息后再来处理
  - 条件足，且有价值，更新到 知识库 或 我的画像汇总，并标注为 achieved
  - 条件足，且无价值，标注为 discorded
